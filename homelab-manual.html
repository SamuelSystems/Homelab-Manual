U1

Installare Ubuntu Server e gestire l'archiviazione

Dalla chiavetta USB al primo login SSH, con una panoramica su partizioni, dischi aggiuntivi e mount permanenti.

Cosa ti serve

  • Un PC, mini-PC o vecchio portatile con almeno 2 GB di RAM e 25 GB di disco (di più è meglio).
  • Una chiavetta USB da almeno 4 GB.
  • La ISO di Ubuntu Server LTS (scaricala da ubuntu.com/download/server). Scegli sempre la versione LTS: riceve aggiornamenti per 5 anni.
  • Un cavo Ethernet: per un server, il Wi-Fi è da evitare quando possibile.

Passo 1 — Crea la chiavetta USB avviabile

Il modo più semplice è usare Balena Etcher o Rufus (Windows): selezioni la ISO, selezioni la chiavetta, avvii. Da Linux/macOS puoi usare direttamente dd:

terminale — creazione USB
# ATTENZIONE: /dev/sdX è la tua chiavetta, verifica con lsblk!
$ sudo dd if=ubuntu-24.04-live-server-amd64.iso of=/dev/sdX bs=4M status=progress
$ sync

Passo 2 — Avvia l'installer

  1. Inserisci la chiavetta nel futuro server e accendilo premendo il tasto per il boot menu (di solito F12, F11, ESC o F2 a seconda del produttore).
  2. Seleziona la chiavetta USB e avvia "Try or Install Ubuntu Server".
  3. Scegli lingua e layout tastiera (Italiano).
  4. Alla schermata di rete, se il cavo Ethernet è collegato riceverai automaticamente un IP via DHCP. Prendi nota di questo IP.

Passo 3 — Configura i dischi (la parte importante)

L'installer propone "Use an entire disk": per iniziare va benissimo. Ti verrà chiesto se usare LVM (Logical Volume Manager): lascialo attivo. LVM ti permette in futuro di ridimensionare le partizioni o aggiungere dischi senza reinstallare.

⚠️

Occhio al default di LVM: Ubuntu di serie usa solo una parte del disco per il volume root. Dopo l'installazione, se vuoi usare il 100% dello spazio, esegui i due comandi qui sotto.

espandere il volume LVM al 100%
$ sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
$ sudo resize2fs /dev/ubuntu-vg/ubuntu-lv

Passo 4 — Utente, SSH e fine installazione

  1. Crea il tuo utente e scegli il nome del server (hostname), es. homelab.
  2. Alla schermata SSH, spunta "Install OpenSSH server": è fondamentale, ti permetterà di gestire il server da remoto.
  3. Salta la selezione degli snap (Docker lo installeremo meglio a mano nel capitolo U4).
  4. A fine installazione rimuovi la chiavetta e riavvia. Da un altro PC ora puoi collegarti così:
primo accesso da remoto
$ ssh tuoutente@192.168.1.50   # usa l'IP annotato prima
💡

D'ora in poi il server può vivere senza monitor né tastiera ("headless"): farai tutto via SSH dal tuo computer principale.

Gestire l'archiviazione: concetti chiave

In Linux i dischi non sono lettere (C:, D:) ma dispositivi in /dev che vengono montati su cartelle. I nomi tipici:

DispositivoCosa è
/dev/sda, /dev/sdbDischi SATA/USB (il primo, il secondo…)
/dev/sda1, /dev/sda2Le partizioni del disco sda
/dev/nvme0n1SSD NVMe (le partizioni sono nvme0n1p1…)
Vedere dischi e spazio
diagnostica dischi
$ lsblk                 # albero di dischi, partizioni e punti di mount
$ df -h                 # spazio usato/libero per ogni filesystem
$ sudo du -sh /var/*    # chi occupa spazio dentro una cartella
$ sudo fdisk -l         # dettagli completi sui dischi
Aggiungere un nuovo disco (es. per i media)

Scenario tipico: colleghi un disco nuovo che appare come /dev/sdb e vuoi usarlo per film, backup o dati dei container.

formattare e montare un disco nuovo
# 1. Crea una partizione (interattivo: g = tabella GPT, n = nuova, w = scrivi)
$ sudo fdisk /dev/sdb

# 2. Formatta la partizione in ext4
$ sudo mkfs.ext4 /dev/sdb1

# 3. Crea il punto di mount e monta
$ sudo mkdir -p /mnt/dati
$ sudo mount /dev/sdb1 /mnt/dati

# 4. Dai i permessi al tuo utente
$ sudo chown -R $USER:$USER /mnt/dati
Montaggio automatico all'avvio: /etc/fstab

Il mount fatto a mano sparisce al riavvio. Per renderlo permanente si aggiunge una riga a /etc/fstab, usando l'UUID del disco (che non cambia mai, a differenza di /dev/sdb):

mount permanente con fstab
# 1. Trova l'UUID della partizione
$ sudo blkid /dev/sdb1
#   → UUID="a1b2c3d4-...."  TYPE="ext4"

# 2. Aggiungi la riga a /etc/fstab
$ sudo nano /etc/fstab
UUID=a1b2c3d4-....  /mnt/dati  ext4  defaults,nofail  0  2

# 3. Testa SENZA riavviare (se non dà errori, sei a posto)
$ sudo mount -a
$ sudo systemctl daemon-reload
⚠️

Un errore in fstab può bloccare l'avvio del server. Usa sempre l'opzione nofail sui dischi dati e testa con sudo mount -a prima di riavviare.

E se voglio RAID o snapshot?

Quando avrai più dischi e dati importanti, valuta filesystem più avanzati: ZFS (integrità dei dati, snapshot, RAID-Z) o mergerfs + SnapRAID (popolare per i media server perché unisce dischi di dimensioni diverse). Per iniziare, ext4 + un backup esterno è più che sufficiente.

Servizi: systemctl

Quasi tutto ciò che gira sul server (SSH, Docker, ecc.) è un servizio gestito da systemd:

$ sudo systemctl stop docker     # ferma
$ sudo systemctl enable docker   # avvio automatico al boot

Utenti e permessi in 60 secondi

Ogni file ha un proprietario, un gruppo e tre livelli di permesso: lettura (r), scrittura (w), esecuzione (x). I casi pratici:

permessi
$ sudo chown -R mario:mario /mnt/dati  # cambia proprietario
$ chmod +x script.sh                   # rende eseguibile uno script
$ chmod 600 chiave.pem                 # leggibile solo dal proprietario
$ sudo usermod -aG docker $USER        # aggiunge il tuo utente al gruppo docker

SSH come un pro

accesso con chiavi (dal tuo PC, non dal server)
# 1. Genera una coppia di chiavi sul TUO computer
$ ssh-keygen -t ed25519

# 2. Copia la chiave pubblica sul server
$ ssh-copy-id tuoutente@192.168.1.50

# 3. Da ora entri senza password. Puoi anche creare un alias:
$ nano ~/.ssh/config
Host homelab
    HostName 192.168.1.50
    User tuoutente

# ...e collegarti semplicemente con:
$ ssh homelab
💡

Una volta che l'accesso con chiave funziona, disattiva il login con password: in /etc/ssh/sshd_config imposta PasswordAuthentication no, poi sudo systemctl restart ssh. È il singolo miglioramento di sicurezza più efficace.

Firewall: ufw

firewall in 5 comandi
$ sudo ufw allow ssh          # PRIMA di attivarlo, o ti chiudi fuori!
$ sudo ufw allow 80,443/tcp   # web (se ti serve)
$ sudo ufw enable             # attiva il firewall
$ sudo ufw status numbered    # regole attive
$ sudo ufw delete 3           # elimina la regola n. 3

U3

Networking, DNS, tunnel e VPN

Come dare al server un indirizzo stabile, dare nomi ai servizi e raggiungere il tuo homelab da fuori casa senza aprire falle di sicurezza.

Le basi: come parla la tua rete

  • IP privato (es. 192.168.1.50): l'indirizzo del server dentro casa, assegnato dal router.
  • IP pubblico: l'indirizzo con cui casa tua si presenta a internet. Spesso cambia (IP dinamico) e con molti operatori è dietro CGNAT (condiviso, non raggiungibile da fuori).
  • Porte: i "canali" di un IP. Un servizio = una porta: SSH è la 22, HTTP la 80, HTTPS la 443.
  • DNS: la rubrica che traduce nomi (jellyfin.casa.lan) in indirizzi IP.

Passo 1 — IP statico per il server

Il server deve avere sempre lo stesso IP. La via più semplice: nel pannello del router cerca "DHCP static lease" (o "prenotazione IP") e associa il MAC address del server a un IP fisso. In alternativa puoi configurarlo sul server con Netplan:

/etc/netplan/50-cloud-init.yaml — IP statico
network:
  version: 2
  ethernets:
    eth0:                        # il nome lo vedi con: ip a
      dhcp4: false
      addresses: [192.168.1.50/24]
      routes:
        - to: default
          via: 192.168.1.1       # il tuo router
      nameservers:
        addresses: [1.1.1.1, 9.9.9.9]

Applica con sudo netplan apply. Se l'editor rompe l'indentazione, ricorda: YAML vuole spazi, mai tab.

Passo 2 — DNS locale: dai un nome ai tuoi servizi

Scrivere 192.168.1.50:8096 stanca subito. Le opzioni, in ordine di impegno:

File hosts

Su ogni PC aggiungi a /etc/hosts (o C:\Windows\System32\drivers\etc\hosts) righe tipo 192.168.1.50 homelab.lan. Semplice ma va fatto su ogni dispositivo.

Pi-hole / AdGuard Home

Un DNS per tutta la rete: blocca la pubblicità e ti fa definire record locali (jellyfin.lan → 192.168.1.50) per tutti i dispositivi in un colpo solo. La scelta consigliata.

💡

Con Pi-hole installato, imposta il suo IP come server DNS nel DHCP del router: ogni dispositivo di casa lo userà automaticamente.

Passo 3 — Accedere da fuori casa

Tre strade, dalla più sicura/semplice alla più "tradizionale":

Opzione A — VPN mesh: Tailscale (consigliata per iniziare)

Tailscale crea una rete privata tra i tuoi dispositivi usando WireGuard. Funziona anche dietro CGNAT, senza aprire porte sul router:

tailscale in 2 comandi
$ curl -fsSL https://tailscale.com/install.sh | sh
$ sudo tailscale up
# apri il link che compare, accedi, fine.
# Installa l'app Tailscale sul telefono/PC: vedrai il server
# con un IP tipo 100.x.y.z raggiungibile ovunque.
Opzione B — VPN self-hosted: WireGuard

Come Tailscale ma tutto in casa tua: veloce, moderno, ma richiede una porta aperta sul router (UDP 51820) e un IP pubblico raggiungibile. Il modo più comodo di gestirlo è il container wg-easy, che ha un'interfaccia web e genera i QR code per i client (lo trovi nel capitolo U5).

Opzione C — Tunnel: Cloudflare Tunnel

Perfetto per esporre siti/servizi web pubblici (es. il tuo blog o Nextcloud) senza aprire nessuna porta: il container cloudflared apre una connessione in uscita verso Cloudflare, che fa da ponte con il tuo dominio. Funziona anche dietro CGNAT e ti dà HTTPS gratis.

SoluzionePorte da aprireFunziona con CGNATIdeale per
TailscaleNessunaSìAccesso personale a tutto il lab
WireGuardUDP 51820NoVPN 100% self-hosted
Cloudflare TunnelNessunaSìServizi web pubblici con dominio
Port forwarding + reverse proxyTCP 80/443NoChi vuole controllo totale
⚠️

Regola d'oro: non esporre mai direttamente su internet SSH, Portainer, o pannelli di amministrazione. Se un servizio non deve essere pubblico, raggiungilo via VPN.

DNS dinamico (DDNS)

Se il tuo IP pubblico cambia ma vuoi comunque raggiungerlo per WireGuard o port forwarding, un client DDNS aggiorna automaticamente un nome a dominio (es. casamia.duckdns.org) con il tuo IP attuale. Servizi gratuiti: DuckDNS, Dynu, oppure l'API di Cloudflare se hai un dominio tuo. Molti router hanno il client DDNS integrato.

Reverse proxy: un solo ingresso, tanti servizi

Invece di ricordare dieci porte diverse, un reverse proxy (Nginx Proxy Manager, Caddy, Traefik) ascolta sulle porte 80/443 e smista in base al nome: jellyfin.tuodominio.it → porta 8096, cloud.tuodominio.it → porta 11000, tutto con certificati HTTPS automatici via Let's Encrypt. Lo trovi tra i servizi del capitolo U5.

U4

Docker & Docker Compose spiegati bene

I container sono il cuore di un homelab moderno: ogni servizio vive isolato, si installa in un minuto e si aggiorna con due comandi. Qui trovi la struttura di un compose file spiegata riga per riga.

Installare Docker (metodo ufficiale)

installazione docker + compose
# Script ufficiale di installazione
$ curl -fsSL https://get.docker.com | sudo sh

# Usa docker senza sudo (poi esci e rientra dalla sessione SSH)
$ sudo usermod -aG docker $USER

# Verifica
$ docker --version
$ docker compose version
$ docker run hello-world

Concetti in 30 secondi

  • Immagine: il "pacchetto" pronto di un'applicazione (es. jellyfin/jellyfin), scaricato da un registry come Docker Hub.
  • Container: un'istanza in esecuzione di un'immagine. Usa e getta: si può distruggere e ricreare senza perdere nulla, se i dati sono nei volumi.
  • Volume: il ponte tra una cartella del server e una cartella dentro il container. È qui che vivono i dati persistenti.
  • Compose: un file YAML (compose.yaml) che descrive uno o più container. Invece di lunghi comandi docker run, scrivi la configurazione una volta e la avvii con docker compose up -d.

Anatomia di un compose file

Questo esempio contiene tutti i blocchi che incontrerai nel 95% dei casi. Leggilo insieme alla spiegazione sotto:

compose.yaml — esempio completo e commentato
services:                          # elenco dei container di questo stack

  jellyfin:                        # nome del servizio (lo scegli tu)
    image: jellyfin/jellyfin:latest  # immagine:tag da Docker Hub
    container_name: jellyfin       # nome fisso del container (opzionale)
    restart: unless-stopped        # riparte da solo dopo crash o riavvii

    ports:                         # ESTERNA:INTERNA
      - "8096:8096"                 # porta 8096 del server → 8096 del container

    volumes:                       # SERVER:CONTAINER
      - ./config:/config            # configurazione (accanto al compose file)
      - /mnt/dati/media:/media:ro   # i tuoi film, in sola lettura (:ro)

    environment:                   # variabili di configurazione
      - TZ=Europe/Rome
      - PUID=1000                   # il tuo utente (vedi: id)
      - PGID=1000

    networks:
      - lab

networks:
  lab:                             # rete interna condivisa dallo stack

Porte: interna vs esterna

La riga "8096:8096" è la fonte di confusione numero uno, quindi vediamola con un disegno:

Il tuo browser192.168.1.50:8096
porta ESTERNA
(host)
Server Ubuntuascolta su :8096
mappata su
porta INTERNA
Containerapp in ascolto su :8096
  • Porta esterna (sinistra): la porta sul server, quella che digiti nel browser. La scegli tu e deve essere unica: due container non possono usare la stessa.
  • Porta interna (destra): la porta su cui l'app ascolta dentro il container. È decisa dall'immagine e non si tocca (la trovi nella documentazione dell'immagine).
  • Conflitto di porta? Cambia solo quella esterna: "8097:8096" → raggiungi il servizio su :8097, ma dentro al container nulla cambia.
  • Un container senza ports: non è raggiungibile dall'esterno, ma gli altri container della stessa rete lo vedono comunque (utile per i database).
  • "127.0.0.1:8096:8096" espone la porta solo in locale sul server: perfetto se davanti c'è un reverse proxy.

Volumi: dove vivono i dati

Anche qui il formato è sinistra:destra = server:container. Due stili:

Bind mount (consigliato)

./config:/config — una cartella reale del server. Sai esattamente dove sono i dati, fai backup con un semplice copia-incolla. Lo standard de facto negli homelab.

Volume gestito

dbdata:/var/lib/postgresql/data — Docker gestisce la cartella per te (in /var/lib/docker/volumes). Va dichiarato nella sezione volumes: di primo livello. Comodo per i database.

  • La regola pratica: se distruggi il container, i dati nei volumi restano. Tutto ciò che non è in un volume sparisce.
  • :ro alla fine = read-only. Usalo per dati che il container deve solo leggere (es. la libreria media).
  • Organizzazione consigliata: una cartella per stack, es. ~/docker/jellyfin/compose.yaml con accanto le sue cartelle di configurazione.

Le altre chiavi che incontrerai

ChiaveA cosa serve

environment:Configurazione via variabili: fuso orario (TZ), password, utente. PUID/PGID (immagini LinuxServer) fanno girare l'app col tuo utente, evitando problemi di permessi sui volumi.

healthcheck:Un comando che verifica se il servizio è davvero vivo.

$ docker compose logs -f        # log in tempo reale (Ctrl+C per uscire)

$ docker system prune           # pulizia immagini/container inutilizzati

Esempio reale: app + database

Uno stack a due servizi (Nextcloud + MariaDB) mostra come i container si parlano tra loro senza esporre il database:

compose.yaml — nextcloud + mariadb
services:
  nextcloud:
    image: nextcloud:latest
    restart: unless-stopped
    ports:
      - "8080:80"              # esterna 8080 → interna 80
    volumes:
      - ./nextcloud:/var/www/html
    environment:
      - MYSQL_HOST=db           # "db" = nome del servizio qui sotto!
      - MYSQL_DATABASE=nextcloud
      - MYSQL_USER=nextcloud
      - MYSQL_PASSWORD=${DB_PASSWORD}  # letta dal file .env
    depends_on:
      - db

  db:
    image: mariadb:11
    restart: unless-stopped   # nessuna porta esposta: solo interno
    volumes:
      - ./db:/var/lib/mysql
    environment:
      - MYSQL_ROOT_PASSWORD=${DB_ROOT_PASSWORD}
      - MYSQL_DATABASE=nextcloud
      - MYSQL_USER=nextcloud
      - MYSQL_PASSWORD=${DB_PASSWORD}
💡

Accanto al compose file crea un file .env con DB_PASSWORD=... e DB_ROOT_PASSWORD=.... Compose lo legge in automatico e le password non finiscono nel file principale.

Esempio pronto: il primo stack

Per partire subito, ecco Portainer + Uptime Kuma. Salva come ~/docker/base/compose.yaml e lancia docker compose up -d:

compose.yaml — stack di partenza
services:
  portainer:
    image: portainer/portainer-ce:latest
    container_name: portainer
    restart: unless-stopped
    ports:
      - "9443:9443"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - ./portainer:/data

  uptime-kuma:
    image: louislam/uptime-kuma:1
    container_name: uptime-kuma
    restart: unless-stopped
    ports:
      - "3001:3001"
    volumes:
      - ./uptime-kuma:/app/data
💡

Poi apri https://192.168.1.50:9443 per Portainer e http://192.168.1.50:3001 per Uptime Kuma. Benvenuto nel tuo homelab. 🎉
